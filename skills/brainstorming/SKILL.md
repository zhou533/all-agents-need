# 将想法转化为设计

通过自然的协作对话，帮助将想法转化为完整的设计和规格说明。

首先了解当前项目上下文，然后逐个提问来细化想法。一旦理解了要构建的内容，呈现设计方案并获得用户批准。

> 在呈现设计方案并获得用户批准之前，**不要**编写任何代码、搭建任何项目或采取任何实施行动。无论项目看起来多简单，这条规则都适用。

## 反模式："这太简单了，不需要设计"

每个项目都要经过这个流程。一个待办列表、一个单函数工具、一次配置变更——都不例外。"简单"的项目恰恰是未经审视的假设导致最多返工的地方。设计可以很简短（对于真正简单的项目只需几句话），但你**必须**呈现设计方案并获得批准。

## 检查清单

按顺序完成以下步骤：

1. **探索项目上下文** — 查看文件、文档、最近的提交记录
2. **提出澄清问题** — 每次一个问题，理解目的/约束/成功标准
3. **提出 2-3 种方案** — 附带权衡分析和你的推荐
4. **呈现设计方案** — 按复杂度分段呈现，每段之后获取用户确认

## 流程图

```
    探索项目上下文
         │
         ▼
    提出澄清问题
         │
         ▼
    提出 2-3 种方案
         │
         ▼
    分段呈现设计方案
         │
         ▼
    用户是否批准？ ──否──▶ 修改后重新呈现
         │
        是
         │
         ▼
      设计完成
```

## 具体流程

**理解想法：**

- 首先了解当前项目状态（文件、文档、最近的提交记录）
- 在提出详细问题之前，先评估范围：如果需求描述了多个独立子系统（例如"构建一个包含聊天、文件存储、计费和分析的平台"），立即指出这一点。不要在一个需要拆解的项目上花时间细化细节。
- 如果项目对于单个规格来说太大，帮助用户拆分为子项目：有哪些独立的部分、它们之间如何关联、应该按什么顺序构建？然后按照正常设计流程对第一个子项目进行头脑风暴。每个子项目有自己独立的设计周期。
- 对于范围适当的项目，逐个提问来细化想法
- 尽可能使用选择题，开放式问题也可以
- 每条消息只问一个问题——如果某个话题需要更深入探索，拆分为多个问题
- 聚焦于理解：目的、约束、成功标准

**探索方案：**

- 提出 2-3 种不同方案及其权衡
- 以对话方式呈现选项，附带你的推荐和理由
- 优先呈现你推荐的方案并解释原因

**呈现设计：**

- 一旦你认为理解了要构建的内容，就呈现设计方案
- 根据复杂度调整每个部分的篇幅：简单的几句话即可，复杂的可到 200-300 字
- 每个部分呈现后询问用户是否满意
- 覆盖：架构、组件、数据流、错误处理、测试
- 准备好在某些内容不清晰时回头澄清

**设计应注重隔离与清晰：**

- 将系统拆分为更小的单元，每个单元有一个明确的职责，通过定义良好的接口通信，并且可以独立理解和测试
- 对于每个单元，你应该能回答：它做什么、如何使用它、它依赖什么？
- 别人能否不看内部实现就理解一个单元的功能？能否修改内部实现而不破坏调用方？如果不能，说明边界需要调整。
- 更小的、边界清晰的单元也更便于 Agent 处理——它在能一次性容纳在上下文中的代码上推理更准确，文件聚焦时编辑更可靠。

**在现有代码库中工作：**

- 在提出变更之前先了解当前结构，遵循已有模式。
- 如果现有代码存在影响当前工作的问题（例如文件过大、边界不清、职责纠缠），将有针对性的改进纳入设计——就像优秀开发者在工作中顺手改进代码一样。
- 不要提出无关的重构。聚焦于服务当前目标。

## 核心原则

- **每次一个问题** — 不要用多个问题让用户应接不暇
- **优先选择题** — 比开放式问题更容易回答
- **严格 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 在确定方案前始终提出 2-3 种方案
- **增量验证** — 呈现设计、获得批准后再继续
- **保持灵活** — 在不清晰时回头澄清
