# Plan 命令

该命令会调用 **planner** agent，在编写任何代码之前先产出一份完整、可执行的实施计划。

## 这个命令会做什么

1. **重述需求** - 明确需要构建什么
2. **识别风险** - 提前暴露潜在问题和阻塞点
3. **创建步骤计划** - 将实施拆分为多个阶段
4. **等待确认** - 在继续前必须收到用户明确操作

## 何时使用

在以下场景使用 `/plan`：
- 开始新功能开发
- 进行较大架构调整
- 执行复杂重构
- 涉及多个文件或组件
- 需求不清晰或存在歧义

## 工作方式

planner agent 将会：

1. **分析请求** 并用清晰语言重述需求
2. **按阶段拆分** 出可执行的具体步骤
3. **识别依赖关系**（组件/模块之间）
4. **评估风险** 和潜在阻塞项
5. **估算复杂度**（高/中/低）
6. **展示计划** 并等待你的明确操作（`modify` 或 `save`）

```
用户：/plan 我需要在市场结算时添加实时通知

Agent（planner）：
# 实施计划：市场结算实时通知

## 需求重述
- 当用户关注的市场结算时，向用户发送通知
- 支持多种通知渠道（站内、邮件、Webhook）
- 确保通知可靠投递
- 通知内容包含市场结果和用户持仓结果

## 实施阶段

### 阶段 1：数据库结构
- 新增 notifications 表，字段：id、user_id、market_id、type、status、created_at
- 新增 user_notification_preferences 表，用于保存渠道偏好
- 在 user_id 和 market_id 上创建索引以提升性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 为失败投递添加重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场结算逻辑（状态变为 "resolved" 时）
- 查询所有在该市场有持仓的用户
- 为每位用户入队通知任务

### 阶段 4：前端组件
- 在头部创建 NotificationBell 组件
- 新增 NotificationList 弹窗
- 通过 Supabase subscriptions 实现实时更新
- 新增通知偏好设置页面

## 依赖项
- Redis（队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件可送达性（需要 SPF/DKIM）
- 中：单市场 1000+ 用户时的性能压力
- 中：市场频繁结算导致通知噪音
- 低：实时订阅带来的额外开销

## 复杂度预估：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：请选择下一步操作（`modify` / `save`）
- `modify`：继续调整计划内容，不进入实现
- `save`：确认当前计划并保存到 `docs/plan/YYYY-MM-DD-{plan功能简称}.md`
```

## 重要说明

**关键**：在你明确选择 `save` 之前，planner agent **不会** 编写任何代码。
`save` 表示你确认当前计划可执行，并同意将其落盘保存。

若你希望修改，请回复：
- `modify: [你的修改内容]`
- `modify: [替代方案]`
- `modify: 先跳过阶段 2，优先做阶段 3`

若你确认保存，请回复：
- `save`

保存规则：
- 遵循planner agent的规范

## 与其他命令的集成
