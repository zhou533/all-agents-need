# Archit 命令

该命令会调用 **architect** agent，进行架构评审、问题讨论、设计提案与技术决策记录。

## 这个命令会做什么

1. **分析范围** - 确定评审对象（crate、模块、功能、全系统）
2. **读取代码** - 通过 Read/Grep/Glob 工具深入分析现有架构
3. **产出交付物** - 由 agent 根据任务类型自主组织输出内容
4. **等待确认** - 涉及文档落盘时必须收到用户明确操作

## 何时使用

在以下场景使用 `/archit`：
- 架构评审：审查现有架构的健康度与问题
- 设计讨论：讨论新功能或重构的架构方案
- 技术决策：评估多个方案的取舍并产出 ADR
- 瓶颈分析：识别可扩展性与性能问题
- 一致性检查：确保新设计与现有架构模式一致

## 工作方式

architect agent 将会：

1. **明确任务类型**：判断是架构评审、设计提案还是 ADR
2. **现状分析**：审查现有架构、识别模式与约定、记录技术债务
3. **需求收集**：整理功能性与非功能性需求、集成点、数据流
4. **产出交付物**：由 agent 根据任务类型自主决定分析深度、组织方式与呈现结构
5. **等待确认**：展示结果并等待用户的明确操作（`modify` 或 `save`）

## 文档落盘规范

architect agent 在用户确认 `save` 后，按仓库现有文档结构、命名约定与任务语义决定保存方式。

常见策略包括：
- 优先复用仓库中已有的架构文档目录与 ADR 约定
- 若仓库已有命名规范，遵循现有规范
- 若仓库没有明确规范，由 agent 选择最合理的位置与文件名，并在保存前说明

## 使用示例

```
用户：/archit xclaw-memory
→ 启动 architect agent 对目标模块进行架构评审
→ 返回：由 agent 自主组织的评审结果与建议

用户：/archit 设计 headless server 模式
→ 启动 architect agent 创建设计提案
→ 返回：由 agent 自主组织的设计方案、取舍与实施建议

用户：/archit ADR: 内存存储应该用 SQLite 还是 sled？
→ 启动 architect agent 产出 ADR
→ 返回：由 agent 自主组织的决策记录、备选方案与理由
```

## 通用能力边界

architect agent 在评审与设计时应优先基于当前仓库的实际结构、约定、技术栈与目标进行判断，而不是依赖预设的项目专属原则。

如仓库中存在以下信息，应主动吸收并纳入分析：
- 现有目录结构与模块边界
- 已有 ADR、设计文档、README 与开发规范
- 代码中已形成的约定、抽象模式与依赖关系
- 性能、安全、扩展性、可维护性等非功能性约束

## 重要说明

**关键**：在你明确选择 `save` 之前，architect agent **不会** 写入任何文件。

若你希望修改，请回复：
- `modify: [你的修改内容]`
- `modify: [替代方案]`

若你确认保存，请回复：
- `save`

## 与其他命令的集成

完成架构设计后：
- 使用 `/plan` 制定详细实施计划
- 使用 `/tdd` 以测试驱动方式实现
- 使用 `/code-review` 做复查
